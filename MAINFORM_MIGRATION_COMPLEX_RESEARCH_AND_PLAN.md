# Комплексное исследование MyManager и план работ

Дата: 2026-03-10  
Основание: анализ текущего состояния кода + `MAINFORM_MIGRATION_MASTER_PLAN.md` + последние коммиты в `master`.

## 1. Цель документа

Сформировать единый, отдельный от master-плана документ, который:
1. Фиксирует текущее состояние программы в целом (архитектура, данные, UX, обработка заказов).
2. Показывает реальные технические риски и узкие места.
3. Дает пошаговый рабочий план: закрыть single-first миграцию и подготовить контролируемый переход к multi-заказам.

## 2. Снимок системы (AS-IS)

### 2.1 Платформа и состав

- Стек: `WinForms`, `.NET 8`, C# (`MyManager.csproj`).
- Точка входа: `Program.cs` -> `Application.Run(new MainForm());`.
- Legacy-формы из `Forms/Archive` уже исключены из сборки (`Compile Remove="Forms\\Archive\\**\\*.cs"`).
- Конфиги/история: JSON (`settings.json`, `history.json`, `pitstop_actions.json`, `imposing_configs.json`).

### 2.2 Карта модулей

| Блок | Назначение | Текущее состояние |
|---|---|---|
| `Forms/MainForm.cs` | Главный UI + orchestration + фильтры + файловые операции + прогресс/статус | Рабочий, но перегружен |
| `Services/OrderProcessor.cs` | Run pipeline (PitStop/Imposing), timeout, progress, multi-item параллелизм | Рабочий |
| `Services/OrderTopologyService.cs` | Нормализация group-first модели, marker `SingleOrder/MultiOrder`, legacy-sync | Рабочий |
| `Models/OrderData.cs` + `OrderFileItem.cs` | Контракт заказа и item-уровня | Рабочий, есть legacy-поля order-level путей |
| `UI/OrderGridContextMenu.cs` | Контекстные действия по колонкам | Single-focus (group пункты убраны) |
| `Forms/*Manager*` + `ConfigService` | Управление сценариями PitStop/Imposing | Рабочий |

### 2.3 Кодовая нагрузка (факты)

- Активный код (без `bin/obj` и без `Forms/Archive`): `33` файла, `9527` строк.
- `MainForm.cs`: `4703` строки (`~49.4%` активной кодовой базы), `266` методов/членов с сигнатурами.
- Сборка baseline на 2026-03-10: `0 warnings`, `0 errors`.

Вывод: программа функционально работает, но имеет высокую концентрацию логики в `MainForm`.

## 3. Состояние миграции (факт относительно master-плана)

### 3.1 Что уже закрыто

1. Старт приложения только через `MainForm`.
2. Legacy `Forms/Archive` исключен из сборки.
3. Контекстное меню и операции по стадиям в новой форме.
4. Run/Stop/Status/Progress интеграция с `OrderProcessor`.
5. StatusStrip-индикаторы (текст, progress, stats, connection, disk, alerts).
6. Nullable-cleanup (сборка без предупреждений).
7. Group-first нормализация (`SingleOrder`/`MultiOrder`) внедрена в pipeline/историю.

### 3.2 Что реально осталось по обязательному контуру

1. Пользовательский фильтр: сейчас используется заглушка (`FilterUsers`) и не подключен к реальному источнику данных.
2. Финальная parity-регрессия `MainForm` против контрольного набора сценариев (single-focus).
3. Дочистка контрактов/нейминга до единой модели `single-order/multi-order` без промежуточных legacy-алиасов в активном коде и документации.

## 4. Ключевые наблюдения и риски

| ID | Приоритет | Наблюдение | Риск |
|---|---|---|---|
| R1 | Высокий | `MainForm` совмещает UI, доменную логику, файловые операции, фильтры, состояние процесса | Сложность изменений, высокий шанс регрессий |
| R2 | Высокий | Фильтр пользователей не связан с фактическими данными заказов | Некорректная фильтрация и ложные ожидания пользователя |
| R3 | Средний | Multi-order backend уже есть, UI временно single-only | Разрыв между возможностями модели и интерфейсом |
| R4 | Средний | Много string-контрактов (имена колонок, статусы, маркеры отображения) | Хрупкость при рефакторинге и локальных правках |
| R5 | Средний | Остатки legacy-подхода: order-level пути плюс item-level пути | Риск расхождений, нуждается в строгой дисциплине нормализации |
| R6 | Средний | Нет автоматических тестов (unit/integration/smoke) | Регрессии выявляются только вручную |
| R7 | Низкий | Есть признаки проблем кодировки в отдельных UI-строках/комментариях | UX-шум и поддерживаемость |

## 5. Целевое решение (TO-BE, прагматичный вариант)

### 5.1 Архитектурная цель

Оставить WinForms и текущую функциональность, но снизить риск изменений за счет декомпозиции `MainForm` и стабилизации контрактов.

### 5.2 Целевые принципы

1. `MainForm` = orchestration + wiring, без дублирования бизнес-логики.
2. Доменная истина по топологии и путям: `OrderTopologyService` + item-уровень.
3. Строгая терминология в коде/доках: `single-order` / `multi-order`.
4. Сначала стабильный single-first контур, затем отдельным этапом полноценный multi-order UI.

### 5.3 Технический целевой контур

1. Разделить `MainForm` по зонам ответственности на partial/модули:
   - Queue+Filters
   - Grid+FileOps
   - RunControl+ProcessorEvents
   - StatusStrip+Indicators
   - History+Persistence
2. Вынести повторяющиеся stage-операции в отдельный helper/service.
3. Ввести единые typed-константы/enum для статусов и колонок (минимум: статусы, stage, column IDs).
4. Подготовить UI-слой для multi-order без включения функционала до завершения single-gates.

## 6. Поэтапный план работ

## Этап A: Закрытие текущей миграции (single-first final)
Срок: 1-2 рабочих дня

1. Подключить реальный источник пользователей и починить user-filter end-to-end.
2. Провести полный ручной регрессионный прогон по чеклисту master-плана.
3. Зафиксировать и закрыть оставшиеся legacy-алиасы именования в активном коде/доках.
4. Обновить `MAINFORM_MIGRATION_MASTER_PLAN.md` по фактическому статусу Go/No-Go.

Definition of Done:
1. User-filter реально фильтрует данные, а не заглушки.
2. Нет блокирующих дефектов по чеклисту single-сценариев.
3. Документально закрыт этап "migration on MainForm (single-first)".

## Этап B: Техническая стабилизация и декомпозиция
Срок: 3-5 рабочих дней

1. Разрезать `MainForm` на логические partial-файлы без изменения поведения.
2. Централизовать stage/file операции (добавление, удаление, rename, copy/move).
3. Убрать string-магии в ключевых контрактах (status/stage/columns).
4. Нормализовать проблемные строки кодировки в UI/сообщениях.

Definition of Done:
1. Поведение идентично baseline, но структура кода модульная.
2. Снижение когнитивной нагрузки при изменениях в фильтрах, статусах, файловых операциях.

## Этап C: Реализация полноценного multi-order UI
Срок: 5-8 рабочих дней

1. Добавить визуальную модель group/item в таблице (expand/collapse, item rows).
2. Добавить item-level действия в UI (без возврата legacy-веток).
3. Реализовать запуск selected items + отображение item progress/status.
4. Добавить сценарии создания/редактирования multi-заказа в формах.

Definition of Done:
1. Multi-order проходит полный цикл от создания до печатного вывода в UI.
2. Single-order остается частным случаем group-first без отдельной архитектуры.

## Этап D: Контроль качества и релизная фиксация
Срок: 2-3 рабочих дня

1. Ввести минимальный smoke-набор (скрипт/чеклист + стабильный dataset).
2. Проверить race/parallel сценарии (`MaxParallelism`, stop/cancel, timeout).
3. Зафиксировать release notes, ограничения и known issues.

Definition of Done:
1. Повторяемый прогон smoke без блокирующих дефектов.
2. Документированная готовность к производственной эксплуатации.

## 7. Приоритет ближайших шагов (что делать следующим)

1. Закрыть Этап A, пункт 1: реальный источник пользователей и рабочий user-filter.
2. Сразу после этого провести полный single-regression (Этап A, пункт 2).
3. Если A закрыт без блокеров, стартовать Этап B (декомпозиция `MainForm`) как обязательную подготовку к multi-order UI.

## 8. Критерии управления риском

1. Любое изменение `MainForm` проходит через ручной smoke по ключевым single-сценариям.
2. Любое изменение модели заказа сначала прогоняется через `OrderTopologyService.Normalize(...)`.
3. Для этапа multi-order не смешивать разработку с "поддержкой legacy UI"; только новая ветка `MainForm`-контур.
