# План миграции на OLV без остановки программы (2026-04-02)

## Стратегия
Миграция по этапам под feature-flag, с постоянным рабочим fallback на текущий `DataGridView`.
Новый OLV-контур разрабатывается и проверяется в отдельном окне прототипа, без отключения текущего рабочего окна.
Перенос в рабочее окно допускается только после полного функционального паритета и отдельного согласования.

## Этап 0. Baseline и guardrails
1. Ввести флаг `UseOlvOrdersGrid`.
2. Зафиксировать baseline:
- время `rebuild-grid`,
- время фильтров,
- UX плавность.
3. Подготовить rollback-процедуру (мгновенный возврат на `dgvJobs`).

Критерий выхода: baseline записан, переключение флага работает.

## Этап 1. Модель и сервисы (без смены UI)
1. Вынести `OrdersGridRowModel` и builder.
2. Вынести фильтрацию/сортировку/counters на модель.
3. Вынести grid-interaction логику в сервис.

Критерий выхода: `dgvJobs` работает через новый слой, поведение не изменено.

## Этап 2. OLV adapter
1. Реализовать `OlvOrdersAdapter`.
2. Подключить `TreeListView` в `pnlTable` как альтернативу.
3. Сделать переключение между `DataGridViewOrdersAdapter` и `OlvOrdersAdapter` по флагу.

Критерий выхода: оба контура запускаются стабильно.

## Этап 3. Функциональный паритет
1. Колонки, значения, формат, статусные иконки.
2. Group/item дерево и expand/collapse.
3. Stage clicks, context menu, drag-drop.
4. Сортировка + индикаторы.
5. Selection sync с tiles.

Критерий выхода: parity matrix без критичных разрывов.

## Этап 4. Тесты
1. Добавить adapter-level тесты.
2. Постепенно ослабить зависимость smoke-тестов от приватного `dgvJobs`.
3. Прогнать stress smoke на больших наборах.

Критерий выхода: green тесты, нет P1/P2 регрессий.

## Этап 5. Rollout
1. Dev/staging: OLV по умолчанию.
2. Production: поэтапное включение.
3. Legacy DataGridView убрать только после стабилизационного окна.

Критерий выхода: OLV основной контур, fallback не нужен.

## Риски и контроль
1. Риск регрессий интеракций: закрывать через parity checklist после каждого этапа.
2. Риск визуальных отличий: закрывать скрин/UX ревью по acceptance checklist.
3. Риск тестового долга: закрывать параллельно, не откладывать на финал.

## Гарантия непрерывности работы
На всем периоде миграции текущая программа и текущий `dgvJobs` остаются рабочими.
OLV вводится параллельно в отдельном окне прототипа и не заменяет рабочее окно до финального решения.
Переключение на новую таблицу в рабочем окне возможно только после:
1. полного паритета функций,
2. прохождения тестов и smoke-проверок,
3. отдельного согласования с владельцем продукта.
