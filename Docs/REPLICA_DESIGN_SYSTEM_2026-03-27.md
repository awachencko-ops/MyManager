# Replica Design System

Дата актуализации: 2026-03-27  
Статус: `Active`

## 1. Назначение

Этот документ фиксирует визуальный язык `Replica`, чтобы новые панели, таблицы, статус-бары, фильтры и служебные блоки выглядели как части одной системы, а не как набор локальных решений.

Документ обязателен как для новых UI-элементов, так и для переработки существующих.

## 2. Визуальный каркас продукта

Текущая UI-модель `Replica` строится на трех уровнях:

1. Узкий левый rail для переключения рабочих областей.
2. Навигационная панель очередей и списков слева от контента.
3. Белая рабочая область с таблицей, тулбаром, фильтрами и нижним статус-баром.

Это не dark UI. Это светлая нейтральная enterprise-система с умеренным контрастом, тонкими разделителями и точечной семантической окраской.

## 3. Базовые токены

### 3.1 Поверхности

- `Dock rail`: `#EEEEF2`
- `Queue / sidebar surface`: `#F5F5F5`
- `Content surface`: `#FFFFFF`
- `Selected row`: `#F3F7FB`
- `Zebra row`: `#FCFDFE`

### 3.2 Разделители

- `Sidebar divider`: `#E0E3E6`
- `Grid line`: `#E7EBF0`
- Все разделители строятся через `1px` line, а не через объемные рамки.

### 3.3 Текст

- `Primary text`: почти черный, высокий контраст
- `Secondary text`: нейтральный серый
- `Tertiary / hint text`: приглушенный серый
- Основной шрифт: `Segoe UI`

### 3.4 Семантические цвета

- Зеленый: только подтвержденное успешное состояние
- Оранжевый: предупреждение или промежуточный процесс
- Красный: ошибка или критичное внимание
- Синий: акцент действия или выбранное состояние

Семантический цвет не должен использоваться как декоративный фон больших зон.

## 4. Пять правил дизайн-кода

### Правило 1. Белый цвет принадлежит данным, а не навигации

Белые поверхности используются для:

- таблиц
- форм
- инспектора
- карточек контента в рабочей области

Белый не используется как отдельная карточка внутри sidebar/queue-зоны, если эта зона уже живет на нейтральной серой поверхности.

### Правило 2. Только плоский UI

Запрещено:

- 3D-рамки
- искусственные тени
- псевдо-объемные карточки
- декоративные градиенты без функциональной роли

Разделение интерфейса строится поверхностями, ритмом и divider-линиями.

### Правило 3. Иконки только из одной семьи

Один экран не смешивает разные визуальные семьи иконок.

Требования:

- единый stroke/weight
- монохромная подача
- цвет иконки меняется только по состоянию
- акцент допускается только для выбранного/ошибочного/активного состояния

### Правило 4. Текст подчиняется иерархии, а не вместимости

Текст делится на три уровня:

1. `Primary`: название сущности или основное действие
2. `Secondary`: роль, категория, подпись
3. `Tertiary`: версия, хинт, системная метка

Если строка не помещается в компактной панели:

- сокращаем формулировку
- переносим полную строку в tooltip
- выносим техническую деталь в status bar / inspector / dialog

Нельзя сохранять длинную техническую строку ценой обрезания композиции.

### Правило 5. Сетка важнее локального удобства

Базовый шаг системы: `8px`.

Стандартные отступы:

- внутренний gutter контейнера: `16px`
- компактный внутренний отступ: `8px`
- крупный визуальный блок не должен собираться на случайных `6/10/13px`

Элемент обязан:

- встраиваться в общую высоту ряда/панели
- иметь предсказуемые поля
- не ломать нижнюю или боковую линию layout

Если блок режется, значит надо упрощать его содержимое или менять сетку, а не “дожимать” текст.

## 5. Layout-правила для Replica

### 5.1 Rail

- ширина стабильная
- только иконки
- hover и active строятся на мягком изменении поверхности
- активное состояние поддерживается простым маркером, а не жирной рамкой

### 5.2 Queue / sidebar

- единая светлая поверхность
- крупные вертикальные элементы списка
- текст выровнен по общей колонке
- счетчики вторичны по контрасту относительно названия

### 5.3 Toolbar

- одна горизонтальная полоса
- actions идут слева направо по приоритету
- иконки и подписи визуально одного веса
- не превращать toolbar в набор разноцветных кнопок

### 5.4 Table / grid

- белая поверхность
- тонкие grid lines
- мягкая zebra-полосатость
- выделение делается светлым tint, без агрессивной заливки
- заголовок таблицы визуально продолжает белую рабочую область

### 5.5 Status bar

- нейтральная база
- основной текст по умолчанию серый/черный
- цвет только у реального статуса
- одновременно активных ярких сигналов должно быть минимум

## 6. Правила для компактных служебных блоков

К компактным блокам относятся:

- `UserPanel`
- connection badges
- мини-статусы
- footer-сводки

Для них действуют отдельные ограничения:

1. Не больше трех строк текста.
2. Не больше одного акцентного действия.
3. Длинные системные строки не показываются целиком внутри блока.
4. Блок должен быть частью поверхности родительской зоны, а не отдельной “карточкой без причины”.

## 7. Текущее нарушение системы: UserPanel

Текущий `UserPanel` нарушает систему по трем причинам:

1. Он выглядит как белая карточка внутри светлого sidebar, хотя по логике должен быть footer-частью той же панели.
2. В нем размещена длинная техническая строка версии, которая визуально режется.
3. Внутренние размеры и отступы не подчинены общей сетке sidebar.

## 8. Целевой стандарт UserPanel

Целевой `UserPanel` должен:

- жить на той же поверхности, что и queue-panel
- отделяться от списка только верхним divider
- содержать avatar, имя, роль и короткую системную строку
- не содержать длинный build hash в явном виде
- выдерживать масштабирование и не резать текст

Рекомендуемая модель:

- высота: `96px`
- внешние отступы: `16px` по горизонтали, `12px` по вертикали
- avatar: `40-44px`
- текстовый блок: 3 строки
- третья строка: короткая, например `v1.0.0` или `Подключен`

## 9. Процесс добавления новых UI-элементов

Перед кодом для любого нового элемента обязателен короткий дизайн-чек.

Нужно ответить на вопросы:

1. В какой зоне системы живет элемент: rail, sidebar, content, tray?
2. Какую поверхность он наследует?
3. Какой у него размер и отступы по сетке `8px / 16px`?
4. Какой у него уровень текста: primary, secondary, tertiary?
5. Нужен ли здесь акцентный цвет, или достаточно нейтральной поверхности?
6. Не дублирует ли он уже существующий визуальный паттерн?

Только после этого пишется код.

## 10. Definition of Done для UI-изменений

UI-изменение считается завершенным только если:

1. элемент соответствует поверхностям и токенам системы;
2. иконки и типографика не выбиваются из существующей семьи;
3. текст не режется на типовом размере окна;
4. блок не выглядит как отдельный стиль внутри существующего layout;
5. решение можно описать через правила системы, а не через “так визуально вроде лучше”.
