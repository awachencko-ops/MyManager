# План миграции основной таблицы на ObjectListView.Repack.Core3 (2026-04-02)

## Цель
Переехать с `DataGridView dgvJobs` на OLV-контрол с сохранением поведения и заметным улучшением отзывчивости.

## Принцип миграции
Поэтапная миграция под feature-flag, без «big bang».

## Этап 0. Подготовка и baseline
1. Добавить пакет:
`<PackageReference Include="ObjectListView.Repack.Core3" Version="2.9.3" />`
2. Зафиксировать baseline-метрики текущего `DataGridView`:
- время `RebuildOrdersGrid` (лог уже есть: `UI-PERF | op=rebuild-grid`),
- время применения фильтров,
- субъективная плавность скролла.
3. Ввести feature-flag `UseOlvOrdersGrid` (настройка + возможность rollback в рантайме).

Критерий выхода: baseline сохранен, флаг работает, сборка зеленая.

## Этап 1. Вынести модель представления таблицы (без смены UI)
1. Создать `OrdersGridRowVm`:
- `RowType` (`Order`/`Item`),
- `OrderInternalId`, `ItemId`,
- поля всех 9 колонок,
- вычисленные флаги (`IsGroupContainer`, `IsMissingSource`, ...).
2. Вынести генерацию строк из `RebuildOrdersGrid()` в отдельный builder-сервис.
3. Вынести stage-операции и resolve-логику из обработчиков в сервис adapter-уровня.

Критерий выхода: текущий DataGridView использует новый row-builder, поведение не меняется.

## Этап 2. Внедрить OLV-контрол параллельно
1. Добавить новый контрол в `pnlTable` рядом с `dgvJobs` (пока скрытый).
2. Создать `OrdersGridAdapter` с двумя реализациями:
- `DataGridViewOrdersAdapter` (текущий),
- `OlvOrdersAdapter` (новый).
3. Подвязать переключение адаптера через feature-flag.

Критерий выхода: приложение запускается в обоих режимах, без падений.

## Этап 3. Колонки и данные OLV
1. Настроить 9 OLV-колонок эквивалентно текущим (`colStatus..colCreated`).
2. Заполнение строк через `SetObjects`/`RefreshObjects` по `OrdersGridRowVm`.
3. Восстановление selection/focus по `OrderInternalId/ItemId`.
4. Реализовать и проверить сортировку:
- initial sort по дате прихода (эквивалент текущего `ArrivalDate DESC`),
- ручное переключение сортировки пользователем + индикатор.

Критерий выхода: данные и сортировка в OLV корректны, визуально паритетно.

## Этап 4. Перенос интеракций
1. Клики/двойные клики -> stage-операции и action-формы.
2. Правый клик -> контекстное меню по колонкам.
3. Drag&Drop между row/column stage.
4. Tooltip для статуса `Ошибка`.
5. Hover-логика и курсор (`Hand/Default`).

Критерий выхода: все ключевые ручные сценарии из DataGridView воспроизводятся в OLV.

## Этап 5. Перенос фильтров и derived-счетчиков
1. Перевести `ApplyStatusFilterToGrid` с UI-row обхода на фильтрацию VM-коллекции.
2. Queue/status/user/date counters считать на модели.
3. Синхронизировать list/tiles selection через единый `SelectionState`, не через прямой обход контролов.

Критерий выхода: фильтры и queue counters совпадают с текущим поведением на тестовых наборах.

## Этап 6. Визуальный паритет
1. Перенести стиль строк и статус-ячейки в OLV renderer.
2. Проверить палитру group-order/item/selected/hover.
3. Сверить UI с текущим чек-листом из дизайн-документа.

Критерий выхода: дизайн не деградировал, команда принимает визуал.

## Этап 7. Тесты и стабилизация
1. Обновить smoke/regression тесты:
- уйти от жесткой зависимости на `DataGridView` приватные поля,
- тестировать через adapter/публичные эффекты.
2. Добавить целевые тесты миграции:
- parity matrix по колонкам,
- sorting/filtering parity,
- group-order expand/collapse,
- drag-drop сценарии.
3. Прогон производительного сценария на больших данных.

Критерий выхода: тесты стабильны, критичные регрессии отсутствуют.

## Этап 8. Rollout и cleanup
1. Включить OLV по умолчанию для dev/staging.
2. После стабилизации убрать fallback-код DataGridView.
3. Удалить мертвые обработчики `dgvJobs` и лишние ветки.

Критерий выхода: OLV — единственный прод-контур таблицы.

## Роллбек-план
1. На любом этапе до cleanup: вернуть `UseOlvOrdersGrid = false`.
2. Если найден критичный баг в OLV-режиме: мгновенно откатить флаг и продолжить работу на текущем DataGridView.

## Что критично не забыть
1. Миграция должна идти через модель данных, а не через копипаст обработчиков между контролами.
2. Нельзя откладывать тестовую адаптацию до конца: тестовый долг здесь высокий.
3. Не трогать tiles-view в первом заходе глубже, чем нужно для синхронизации selection.
