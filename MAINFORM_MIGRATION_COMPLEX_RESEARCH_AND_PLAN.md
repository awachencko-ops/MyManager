# Комплексное исследование MyManager и план работ

Дата: 2026-03-11  
Основание: анализ текущего состояния кода + `MAINFORM_MIGRATION_MASTER_PLAN.md` + фактические изменения структуры `MainForm`.

## 1. Цель документа

Сформировать единый, отдельный от master-плана документ, который:
1. Фиксирует текущее состояние программы в целом (архитектура, данные, UX, обработка заказов).
2. Показывает реальные технические риски и узкие места.
3. Дает пошаговый рабочий план: закрыть single-first миграцию и подготовить контролируемый переход к multi-заказам.

2. Legacy `Forms/Archive` исключен из сборки.
3. Контекстное меню и операции по стадиям в новой форме.
4. Run/Stop/Status/Progress интеграция с `OrderProcessor`.

8. Крупная декомпозиция `MainForm` выполнена: partial-файлы + структурирование по папкам.

### 3.2 Что реально осталось по обязательному контуру

3. Дочистка string-контрактов (status/stage/column-name) до централизованного typed-контура.
4. Подключение локальной сети в минимально-полезном режиме (см. раздел 9).

## 4. Ключевые наблюдения и риски

| ID | Приоритет | Наблюдение | Риск |
|---|---|---|---|
| R1 | Средний | `MainForm` всё ещё крупный как aggregate (хотя монолит уже разрезан) | Сложность будущих изменений, если не продолжить детализацию |
| R2 | Высокий | Фильтр пользователей не связан с фактическими данными заказов | Некорректная фильтрация и ложные ожидания пользователя |

## 5. Целевое решение (TO-BE, прагматичный вариант)

### 5.1 Архитектурная цель

Оставить WinForms и текущую функциональность, но довести модульность до состояния, когда изменения не требуют правок в большом числе разнородных файлов.

### 5.2 Целевые принципы

1. `MainForm` = orchestration + wiring, без дублирования бизнес-логики.
2. Доменная истина по топологии и путям: `OrderTopologyService` + item-уровень.
3. Строгая терминология в коде/доках: `single-order` / `multi-order`.
4. Сначала стабильный single-first контур, затем отдельным этапом полноценный multi-order UI.
5. LAN-интеграция вводится поэтапно: от минимального контроля доступности к полноценному источнику данных.

### 5.3 Технический целевой контур

1. Довести декомпозицию `MainForm` до мелких стабильных модулей.
2. Централизовать stage/file операции в выделенный helper/service (без дублирования между UI-сценариями).
3. Ввести typed-константы/enum для статусов, stage, column IDs.
4. Подготовить UI-слой для multi-order без включения функционала до завершения single-gates.
5. Зафиксировать минимальный LAN baseline как отдельный deliverable.

## 6. Поэтапный план работ

## Этап A: Закрытие текущей миграции (single-first final)
Срок: 1-2 рабочих дня

1. Подключить реальный источник пользователей и починить user-filter end-to-end.
2. Провести полный ручной регрессионный прогон по чеклисту master-плана.
3. Зафиксировать и закрыть оставшиеся legacy-алиасы именования в активном коде/доках.
4. Обновить `MAINFORM_MIGRATION_MASTER_PLAN.md` по фактическому статусу Go/No-Go.

Definition of Done:
1. User-filter реально фильтрует данные, а не заглушки.
2. Нет блокирующих дефектов по чеклисту single-сценариев.
3. Документально закрыт этап "migration on MainForm (single-first)".

## Этап B: Техническая стабилизация и декомпозиция
Срок: 3-5 рабочих дней

1. Разрезать `MainForm` на логические partial-файлы без изменения поведения. (выполнено в крупном объеме)
2. Централизовать stage/file операции (добавление, удаление, rename, copy/move). (in progress)
3. Убрать string-магии в ключевых контрактах (status/stage/columns). (pending)
4. Нормализовать проблемные строки кодировки в UI/сообщениях. (in progress)

Definition of Done:
1. Поведение идентично baseline, но структура кода модульная.
2. Снижение когнитивной нагрузки при изменениях в фильтрах, статусах, файловых операциях.

## Этап B1: LAN baseline (минимальный)
Срок: 0.5-1 рабочий день

1. Зафиксировать рабочие UNC-пути в настройках (`OrdersRootPath`, `TempFolderPath`, `GrandpaPath`).
2. Оставить/подтвердить индикатор доступности сети через `toolConnection` (проверка доступа к корневому пути).
3. Добавить явный критерий offline-режима: при недоступности пути UI не падает, операции дают понятный статус.

Definition of Done:
1. При отключении/включении сети индикатор меняется в пределах таймера.
2. Приложение продолжает работать в автономном режиме без crash.

## Этап C: Реализация полноценного multi-order UI
Срок: 5-8 рабочих дней

1. Добавить визуальную модель group/item в таблице (expand/collapse, item rows).
2. Добавить item-level действия в UI (без возврата legacy-веток).
3. Реализовать запуск selected items + отображение item progress/status.
4. Добавить сценарии создания/редактирования multi-заказа в формах.

Definition of Done:
1. Multi-order проходит полный цикл от создания до печатного вывода в UI.
2. Single-order остается частным случаем group-first без отдельной архитектуры.

## Этап D: Контроль качества и релизная фиксация
Срок: 2-3 рабочих дня

1. Ввести минимальный smoke-набор (скрипт/чеклист + стабильный dataset).
2. Проверить race/parallel сценарии (`MaxParallelism`, stop/cancel, timeout).
3. Зафиксировать release notes, ограничения и known issues.

Definition of Done:
1. Повторяемый прогон smoke без блокирующих дефектов.
2. Документированная готовность к производственной эксплуатации.

## 7. Приоритет ближайших шагов (что делать следующим)

1. Закрыть Этап A, пункт 1: реальный источник пользователей и рабочий user-filter.
2. Запустить Этап B1 (LAN baseline) как быстрый operational gate.
3. Провести полный single-regression (Этап A, пункт 2).
4. Дожать Этап B (typed-контракты + финальная чистка кодировки).

## 8. Критерии управления риском

1. Любое изменение `MainForm` проходит через ручной smoke по ключевым single-сценариям.
2. Любое изменение модели заказа сначала прогоняется через `OrderTopologyService.Normalize(...)`.
3. Для этапа multi-order не смешивать разработку с "поддержкой legacy UI"; только новый `MainForm`-контур.
4. Для LAN-фич фиксировать fallback-поведение в offline-режиме до релиза.

## 9. Подключение к локальной сети: варианты

### Вариант 1 (базовый, "для галочки") — рекомендован сейчас

1. Использовать UNC-пути в настройках (`\\\\server\\\\share\\\\...`) для рабочих папок.
2. Проверять доступность по существующему индикатору `toolConnection`.
3. В случае недоступности пути работать в автономном режиме с явным статусом `Сервер: автономно`.

Плюсы: быстро, почти без новых рисков, уже частично есть в коде.  
Минусы: это контроль доступности, а не полноценная сетевой синхронизация пользователей/данных.

### Вариант 2 (практичный)

1. Хранить `users.json`/`users.csv` на сетевой шаре.
2. Загружать список пользователей при старте и периодически обновлять.
3. Держать локальный кэш на случай недоступности сети.

Плюсы: решает текущую задачу по user-filter.  
Минусы: нужен контроль конфликтов/версий и fallback-логика.

### Вариант 3 (зрелый)

1. Центральный сервис/API или БД.
2. Авторизация, аудит, централизованная очередь/статусы.

Плюсы: масштабируемо и управляемо.  
Минусы: высокий порог внедрения и сроки.

Итог по текущему этапу: ставим **Вариант 1** как базовый gate + планируем **Вариант 2** для реального закрытия user-filter.
