# План реализации: обработка групп файлов в одном заказе

## 1. Цель
Сконсолидировать реализацию многофайлового заказа в единую стратегию:
- заказ может содержать группу файлов;
- каждый файл проходит независимый pipeline (`исходный -> подготовка -> печать`);
- статус заказа агрегируется по статусам файлов;
- поддерживаются массовые операции и частичная готовность.

## 2. Ключевые UX-правила

### 2.1. Представление заказа-группы в таблице
- Верхний уровень таблицы — строка заказа (group header row).
- Для заказа-группы слева добавляется **чекбокс/переключатель раскрытия**:
  - включён/активен -> список файлов внутри группы раскрыт;
  - выключен -> список файлов скрыт.
- В развёрнутом виде под строкой группы показываются строки файлов (items).

### 2.2. Поведение при преобразовании заказа в группу
При действии **«Преобразовать в группу»**:
1. Текущие данные одиночного заказа переносятся в первый `OrderFileItem`.
2. Заказ становится контейнером `Items`.
3. Основная строка группы (header row) после преобразования содержит:
   - номер заказа;
   - операции для редактирования (PitStop/Imposing и связанные действия).
4. Эти операции в header row **редактируемые** и применяются **ко всем файлам группы**.

### 2.3. Рабочая строка добавления
- Внизу списка файлов группы всегда есть пустая рабочая строка.
- При добавлении файла в рабочую строку:
  - создаётся новый `OrderFileItem`;
  - автоматически добавляется новая пустая рабочая строка.

## 3. Модель данных

### 3.1. `OrderData`
- Добавить `List<OrderFileItem> Items`.
- Сохранить legacy-поля `SourcePath`, `PreparedPath`, `PrintPath` для миграции/совместимости.
- Добавить/использовать метод пересчёта агрегированного статуса заказа.

### 3.2. `OrderFileItem`
Минимальные поля:
- идентификатор `ItemId`;
- клиентская метка файла `ClientFileLabel`;
- вариант `Variant` (значение из справочника);
- пути стадий `SourcePath`, `PreparedPath`, `PrintPath`;
- технические вложения `TechnicalFiles`;
- статус `FileStatus`, причина `LastReason`, `UpdatedAt`;
- порядок `SequenceNo`.

### 3.3. Глобальные настройки (`AppSettings`)
- глобальная сортировка файлов в заказе;
- справочник вариантов `VariantDictionary`;
- флаг разрешения ручной перестановки (drag-and-drop);
- `MaxParallelism` (0 = без лимита);
- `AutoRenameOnDuplicate`.

## 4. Правила статусов

### 4.1. На уровне файла
- `⚪ Ожидание`
- `🟡 В работе`
- `✅ Готово`
- `🔴 Ошибка`

### 4.2. На уровне заказа (агрегирование)
- все `Ожидание` -> `⚪ Ожидание`;
- все `Готово` -> `✅ Готово`;
- все `Ошибка` -> `🔴 Ошибка`;
- есть хотя бы один `В работе` -> `🟡 В работе (X/Y)`;
- смешанный результат с успехами/ошибками -> `⚠ Частично готово (X/Y)`.

## 5. Обработка и batch

### 5.1. Режим запуска
Перед запуском выбрать:
- обработать все файлы группы (по умолчанию);
- обработать выбранные файлы.

### 5.2. Применение операций из строки группы
- Изменения операций в header row — это массовое изменение настроек для всех `Items` группы.
- Для новых добавленных файлов группы применяются текущие операции группы по умолчанию.

### 5.3. FIFO и ручная перестановка
- Базовый порядок выполнения — `SequenceNo`.
- Разрешена ручная перестановка строк файлов, при этом `SequenceNo` пересчитывается.

### 5.4. Параллелизм
- Выполнение по файлам с учётом `MaxParallelism`.
- При `MaxParallelism=0` использовать максимально доступный параллелизм среды.

## 6. Дубликаты имён
- В пределах одной стадии дубликаты обрабатываются автопереименованием (`_1`, `_2`, ...).
- Проверка и нормализация имён выполняется до фактической записи/копирования.

## 7. Миграция истории

### 7.1. Стратегия
1. При первом запуске новой схемы — backup `history.json.bak`.
2. Для legacy-заказа создать `Items[0]` из старых полей путей.
3. Пересчитать агрегированный статус.
4. Сохранить в новом формате.

### 7.2. Безопасность
- При ошибке миграции: откат к backup и уведомление пользователя.

## 8. Этапы внедрения

### Этап 1 — Data/Storage
- Ввести `OrderFileItem`, `Items`, настройки, миграцию, агрегатор статусов.

### Этап 2 — UI групп
- Добавить строку группы с левым переключателем раскрытия/сворачивания.
- Сделать редактируемыми на строке группы операции, применяемые ко всем файлам группы.
- Отрисовать строки файлов и рабочую строку добавления.

### Этап 3 — Pipeline и batch
- Перевести обработчик на `Items`.
- Реализовать запуск все/выбранные.
- Добавить массовые операции и прогресс `X/Y`.

### Этап 4 — Стабилизация
- Оптимизация для больших групп (50+ файлов).
- Тесты: FIFO, дубликаты, частичные успехи, массовое применение операций группы.

## 9. Критерии готовности
- Заказ успешно преобразуется в группу без потери данных.
- Строка группы показывает номер и редактируемые операции.
- Операции строки группы применяются ко всем файлам группы.
- Левый переключатель корректно разворачивает/сворачивает список файлов.
- История старого формата мигрируется безопасно с backup.
- Поддерживается частичный статус `⚠ Частично готово (X/Y)`.
