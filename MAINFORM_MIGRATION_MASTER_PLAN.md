# MASTER PLAN: Полный перенос MyManager на MainForm

Дата фиксации плана: 2026-03-10  
Период исполнения (базовый): 2026-03-09 — 2026-03-15  
Статус обновлен: 2026-03-10  
Главная цель: перевести рабочий контур с legacy `Form1` на `MainForm` с подтвержденным паритетом ключевых сценариев.

## 1. Цель и Definition of Done

Полный перенос считается достигнутым, если одновременно выполнено:
1. Все сценарии основной таблицы работают из `MainForm` без обращения к legacy-обработчикам `Form1`.
2. Запуск/остановка обработки заказа, смена статусов и отображение логов выполняются из `MainForm`.
3. Контекстное меню, Drag&Drop и файловые операции по стадиям работают в `MainForm`.
4. Фильтры (`Состояние задания`, `Номер заказа`, `Пользователь`, `Дата поступления`, `Начало обработки`) стабильно работают без конфликтов popup-календарей/overlay.
5. Пройдена ручная регрессия на новой форме без блокирующих дефектов.

## 1.1 Критически важный принцип: group-first

1. Базовая модель всегда: `1 заказ = группа файлов (Items)`.
2. Схема `1 заказ = 1 файл` не отдельная архитектура, а частный случай group-first (ровно один `item`).
3. Для явного разделения используется маркер:
   - `SingleOrder` — один `item` в заказе.
   - `MultiOrder` — два и более `item`.
4. Legacy-поля `SourcePath/PreparedPath/PrintPath` сохраняются только для обратной совместимости и синхронизируются с `item`-уровнем через нормализатор.

## 1.2 Конфликты и принятое решение

| Потенциальный конфликт | Риск | Решение |
|---|---|---|
| В заказе есть пути на уровне order, но нет `Items` | Потеря файлов при переходе к групповой модели | Автонормализация: создается synthetic `item`, пути переносятся в него |
| В single-заказе `order.PrintPath` и `item.PrintPath` расходятся | Непредсказуемая обработка/копирование | Приоритет существующего файла, затем непустого пути; после выбора значения синхронизируются |
| Разные части кода по-разному определяют «группу» | Регрессии UI и обработки | Единый `OrderTopologyService.IsMultiOrder(...)` |
| Маркер single/multi не совпадает с фактическим `Items` | Неверная маршрутизация сценариев | Нормализатор пересчитывает маркер при сохранении/запуске |
| История содержит null/битые item | Падения в grid/processor | Нормализатор очищает null-item и восстанавливает `ItemId`, `SequenceNo`, `UpdatedAt` |

1. Один день = один фокусный блок переноса.
2. После каждого блока обязательна ручная регрессия по чеклисту.
3. Любая блокирующая регрессия возвращает блок в `In progress`.
4. `Form1` хранится в `Forms/Archive/` как reference, но исключен из сборки; тестирование ведется только на `MainForm`.
5. Все статусы работ фиксируются в этом документе.
