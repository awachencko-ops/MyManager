# MyManager: групповые заказы (group-order) — анализ рисков, проектирование и план внедрения (с адаптацией под PostgreSQL)

Дата актуализации: 2026-03-13
Статус: проектный документ (без изменений в коде)

## 1. Контекст и цель

Документ фиксирует целевую логику:
1. **Заказ = контейнер**.
2. Контейнер хранит ключевые поля менеджера:
   - `OrderNumber` (номер заказа);
   - `ManagerOrderDate` (дата оформления/просчёта менеджером).
3. Внутри контейнера находятся **файлы (items)**.
4. Для каждого item отдельно фиксируется `PrepressArrivalAt` (дата поступления в препресс).

Цель: стандартизировать модель «контейнер + файлы», утвердить UI-правила для group-order и выбрать корректную архитектуру доступа к PostgreSQL в локальной сети.

## 2. Терминология (обновлено)

1. `single-order` — заказ-контейнер, в котором ровно 1 item.
2. `group-order` — заказ-контейнер, в котором 2+ items.
3. Термин `multi-order` в новых спецификациях не используем; заменяем на `group-order`.

## 3. Что уже есть в текущем проекте

1. `OrderData` уже хранит контейнер `Items` и разделяет `OrderDate` / `ArrivalDate`.
2. `OrderFileItem` уже хранит стадии, статус, операции, `SequenceNo`.
3. В текущем коде есть маркер топологии `SingleOrder/MultiOrder`; на следующем шаге его нужно семантически переименовать в документации и UI на `single/group`.

Итог: фундамент готов, нужно дожать единые контракты и исключить двусмысленность.

## 4. Целевая доменная модель

### 4.1 `Order` (контейнер)

Обязательные бизнес-поля:
1. `OrderNumber`.
2. `ManagerOrderDate`.

Служебные поля:
1. `OrderId` (UUID).
2. `CreatedAt`, `UpdatedAt`.
3. `CreatedByUser`, `UpdatedByUser` (рекомендуется для LAN).
4. Агрегаты: `ItemsCount`, `AggregateStatus`.

### 4.2 `OrderItem` (файл)

1. `ItemId` (UUID), `OrderId` (FK).
2. `PrepressArrivalAt`.
3. `SequenceNo` (стабильный порядок внутри контейнера).
4. `SourcePath`, `PreparedPath`, `PrintPath`.
5. `ItemStatus`, `LastReason`, `UpdatedAt`.
6. `PitStopAction`, `ImposingAction`.

### 4.3 Маркировка для интерфейса — обязательна

По решению: **метки нужны** для UI.

1. В интерфейсе явно показываем тип контейнера:
   - `single-order`;
   - `group-order`.
2. Это влияет на визуал и поведение, поэтому метка — не декоративная, а функциональная.
3. В источнике истины метка вычисляется из `ItemsCount`, но в UI отображается как отдельный признак/бейдж.

## 5. UI/UX-правила для group-order (зафиксировано)

1. Строка `group-order` имеет **отличающийся цвет плашки** (от single-order).
2. По **ЛКМ** на строке контейнера выполняется **expand/collapse** списка файлов.
3. В строке контейнера показываем только контейнерные поля:
   - номер заказа;
   - дата оформления менеджером.
4. В дочерних строках файлов (items) показываем:
   - всю операционную информацию файла;
   - при необходимости дублируем контейнерный контекст (номер/дата), чтобы item можно было читать автономно в списке/фильтрах.
5. Кнопка в toolbar: **«Добавить файл в заказ»** — штатная команда добавления item в выбранный контейнер.

## 6. Риски и контрмеры

### R1. Путаница двух дат
Риск: смешение `ManagerOrderDate` и `PrepressArrivalAt`.
Контрмеры:
1. Раздельные подписи/колонки/подсказки.
2. Явные имена в DTO/БД.
3. Валидация на уровне API/репозитория.

### R2. Расхождение order-level и item-level путей
Риск: legacy поля контейнера конфликтуют с item-путями.
Контрмеры:
1. Источник истины — item-level пути.
2. Legacy-поля только как переходный адаптер.
3. План удаления legacy-режима отдельным этапом.

### R3. Конкурентные изменения в LAN
Риск: несколько операторов редактируют одну запись.
Контрмеры:
1. Optimistic concurrency (`version`, 409 Conflict).
2. Аудит изменений (`order_events`).
3. Идемпотентные команды изменения.

### R4. Нестабильный порядок файлов
Риск: разный порядок items на разных рабочих местах.
Контрмеры:
1. `SequenceNo` + `unique(order_id, sequence_no)`.
2. Явная команда reorder.

### R5. Недоопределённость архитектуры доступа к PostgreSQL
Риск: прямые подключения клиентов к БД без единой бизнес-валидации.
Контрмеры:
1. Утвердить архитектурный профиль до начала кодинга.
2. Зафиксировать один «источник бизнес-правил» (API-сервис или единый backend-процесс).

## 7. Нужен ли API? Варианты и рекомендация

Ниже — практический разбор вашей идеи «мой ПК как сервер, база всегда смотрит за всем».

### Вариант A: клиенты ходят в PostgreSQL напрямую
Плюсы:
1. Быстрее старт на раннем прототипе.
2. Меньше компонентов.

Минусы:
1. Бизнес-правила размазываются по клиентам.
2. Сложнее контроль версий/конкурентности.
3. Сложнее аудит и безопасные миграции.
4. Более высокий риск «тихих» рассинхронов в group-order логике.

### Вариант B: PostgreSQL + API-сервис (рекомендуется)
Плюсы:
1. Единая точка бизнес-логики (контейнер/item, метки, expand-модель, валидации дат).
2. Нормальная обработка конкурентности (`version`, конфликты, ретраи).
3. Удобно журналировать и мониторить события.
4. Проще развивать UI-клиенты в будущем.

Минусы:
1. Плюс один процесс (service/API).
2. Нужно поддерживать деплой API рядом с БД.

### Вариант C: гибрид для мягкого старта
1. Сначала единый backend-процесс (локальный сервис на «серверном ПК»), который уже реализует API-контракты.
2. Клиенты общаются только с ним.
3. При необходимости позже выносится в полноценный сетевой API без изменения доменной модели.

### Практический вывод
Если цель — «чтобы работало как часы» в LAN с несколькими операторами, **API-слой нужен** (хотя бы минимальный внутренний service API). Прямое подключение всех клиентов к таблицам PostgreSQL допустимо только как очень временный этап, иначе риски консистентности и сопровождения резко растут.

## 8. PostgreSQL: целевой контракт (черновой)

### 8.1 Таблицы

1. `orders`
   - `order_id uuid pk`
   - `order_number text not null`
   - `manager_order_date date not null`
   - `created_at timestamptz not null`
   - `updated_at timestamptz not null`
   - `aggregate_status text not null`
   - `items_count int not null default 0`
   - `version bigint not null default 0`

2. `order_items`
   - `item_id uuid pk`
   - `order_id uuid not null references orders(order_id) on delete cascade`
   - `prepress_arrival_at timestamptz not null`
   - `sequence_no bigint not null`
   - `source_path text not null default ''`
   - `prepared_path text not null default ''`
   - `print_path text not null default ''`
   - `item_status text not null`
   - `last_reason text not null default ''`
   - `pitstop_action text not null default '-'`
   - `imposing_action text not null default '-'`
   - `updated_at timestamptz not null`
   - `version bigint not null default 0`

3. `order_events`
   - `event_id uuid pk`
   - `order_id uuid not null`
   - `item_id uuid null`
   - `event_type text not null`
   - `payload jsonb not null`
   - `actor text not null`
   - `created_at timestamptz not null`

### 8.2 Ограничения и индексы

1. `unique(order_id, sequence_no)`.
2. Индексы: `order_items(order_id)`, `orders(updated_at)`, `orders(manager_order_date)`.
3. `check(sequence_no > 0)`.
4. Отдельно утвердить правило уникальности `order_number` (глобально или в паре с датой).

## 9. Минимальный API-контракт (если идём по recommended пути)

1. `POST /orders` — создать контейнер.
2. `PATCH /orders/{id}` — обновить номер/дату менеджера.
3. `POST /orders/{id}/items` — добавить файл в заказ.
4. `PATCH /orders/{id}/items/{itemId}` — обновить item.
5. `POST /orders/{id}/items/reorder` — reorder.
6. `GET /orders` — список (включая признак `single-order/group-order`).
7. `GET /orders/{id}` — контейнер + items.
