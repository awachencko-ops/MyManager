# План реализации: обработка групп файлов в одном заказе

## 1) Цель и рамки
Сделать полноценную поддержку групп файлов в заказе без потери обратной совместимости:
- заказ может содержать 1..N файлов;
- каждый файл проходит независимый pipeline `исходный -> подготовка -> печать`;
- на уровне заказа отображается агрегированный статус (`X/Y`);
- операции из основной строки группы можно редактировать и массово применять ко всем файлам группы.

Не входит в MVP:
- изменение формата action-конфигов;
- переписывание всей таблицы на новый UI-фреймворк.

---

## 2) UX-спецификация (обновлено)

### 2.1. Основная строка группы (Group Header Row)
После «Преобразовать в группу» основная строка заказа содержит только:
1. **переключатель слева** (галочка/chevron) для раскрытия и сворачивания;
2. **номер заказа**;
3. **редактируемые операции** (`PitStop`, `Imposing`, связанные действия).

Важно:
- Редактирование операций в основной строке применяется **ко всем файлам группы**.
- Для новых файлов, добавленных в эту группу, операции группы подставляются как значения по умолчанию.

### 2.2. Левый переключатель раскрытия/сворачивания
- В свернутом состоянии отображается только основная строка группы.
- В раскрытом состоянии под ней показываются строки файлов группы.
- Состояние раскрытия сохраняется в runtime (словарь `InternalId -> isExpanded`) и восстанавливается после обновления таблицы.

### 2.3. Строки файлов и рабочая строка
- Для каждого файла — отдельная строка item.
- Внизу раскрытой группы всегда есть рабочая пустая строка (drop target).
- При добавлении файла в рабочую строку:
  - создается `OrderFileItem`;
  - автоматически добавляется новая пустая рабочая строка.

### 2.4. Преобразование одиночного заказа в группу
Команда контекстного меню: **«Преобразовать в группу»**.
- Если у заказа есть legacy-пути (`Source/Prepared/Print`) и нет `Items`, создается `Items[0]`.
- Основная строка переключается в режим группы (см. 2.1).
- Группа раскрывается автоматически после преобразования.

## 4) Правила выполнения

### 4.1. Запуск обработки
Диалог/меню перед стартом:
1. Обработать все файлы (по умолчанию)
2. Обработать только выбранные

### 4.2. Порядок и перестановка
- Базовый порядок — `SequenceNo` (FIFO).
- Разрешено ручное изменение порядка drag-and-drop.
- После ручной перестановки выполняется пересчет `SequenceNo`.

### 4.3. Параллельность
- Обработка items параллельно с учетом `MaxParallelism`.
- `MaxParallelism = 0` трактуется как «без ограничения пользователем».

### 4.4. Дубликаты
- В пределах одной стадии используется автопереименование (`_1`, `_2`, ...).
- Проверка имени выполняется до копирования/перемещения.

## 5) Статусы

### 5.1. Статус файла
- `⚪ Ожидание`
- `🟡 В работе`
- `✅ Готово`
- `🔴 Ошибка`

### 5.2. Агрегированный статус заказа
- все `Ожидание` -> `⚪ Ожидание`
- все `Готово` -> `✅ Готово`
- все `Ошибка` -> `🔴 Ошибка`
- есть хотя бы один `В работе` -> `🟡 В работе (X/Y)`
- смешанный итог -> `⚠ Частично готово (X/Y)`

---

## 6) Миграция истории

1. На первом запуске новой схемы создать `history.json.bak`.
2. Для legacy-заказов без `Items` создать `Items[0]` из старых путей.
3. Пересчитать статус заказа через агрегатор.
4. Сохранить в новом формате.
5. При ошибке — откат к backup + уведомление пользователя.

## 8) Критерии приемки
1. Одиночный заказ конвертируется в группу без потери путей.
2. Основная строка группы содержит номер + редактируемые операции.
3. Изменение операций в основной строке применяется ко всем файлам группы.
4. Левый переключатель корректно разворачивает/сворачивает список файлов.
5. Рабочая строка всегда присутствует внизу раскрытой группы.
6. Миграция старой истории работает с backup и безопасным откатом.
7. Частичный результат всегда показывает `⚠ Частично готово (X/Y)`.

## 9) Риски и меры снижения

1. **Сложность иерархии в `DataGridView`**
   - Риск: несоответствие индексов строк заказам/item после сортировки и refresh.
   - Мера: ввести внутренний row-view-model (`OrderRow`, `ItemRow`, `DraftRow`) и рендерить grid только из него.

2. **Гонки статусов при параллельной обработке**
   - Риск: «скачущий» агрегированный статус и неконсистентные перерисовки.
   - Мера: централизованный `RefreshAggregatedStatus()` + UI-обновления только через `Invoke/BeginInvoke`.

3. **Проблемы миграции истории**
   - Риск: частично преобразованные данные при сбое в середине процесса.
   - Мера: миграция в память, затем атомарная запись; обязательный `.bak` до старта миграции.

4. **Просадка производительности на 50+ файлов**
   - Риск: подвисание интерфейса при частых перерисовках.
   - Мера: batched refresh grid, обновление только затронутых строк, ограничение частоты status-update событий.
