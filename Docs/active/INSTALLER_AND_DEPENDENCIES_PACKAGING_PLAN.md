# Этап 5: установочный пакет и упаковка зависимостей (отдельный документ)

Дата актуализации: 2026-03-20
Статус: Planned (выделен как самостоятельный документ этапа 5)

## 0. Граница документа

Этот файл фиксирует только этап 5: installer, упаковку runtime-зависимостей и первичную установку на новый ПК.
Этап 4 (EF/API/auto-update) ведётся отдельно в `Docs/ready/4_EF_MIGRATIONS_API_AND_AUTOUPDATE_ROLLOUT_PLAN.md`.

## 1. Цель этапа

Подготовить единый установочный пакет для развёртывания Replica на новых ПК без ручной настройки по офису.

Ключевая идея:
1. Один инсталлятор ставит клиент.
2. Инсталлятор проверяет/устанавливает обязательные зависимости.
3. После установки клиент дальше обновляется через автообновление (этап 4), без повторных ручных инсталляций.

## 2. Что входит в установку

### 2.1 Обязательное

1. `Replica.Client` (основное приложение).
2. Конфигурация подключения к API (`BaseUrl`, например `http://<LAN-IP>:5000`).
3. Ярлыки (Desktop/Start Menu).
4. Логи и рабочие папки (создание структуры при установке).

### 2.2 Зависимости/пререквизиты (проверка + установка)

1. `.NET Runtime` нужной версии (или self-contained стратегия).
2. `Ghostscript` (если реально используется в pipeline и обязательна на клиентском ПК).
3. VC++ Redistributable (если нужен для конкретных библиотек).
4. Дополнительные внешние утилиты, которые использует обработка PDF/печати (по итоговому audit зависимостей).

> Правило: включаем в инсталлятор только подтверждённые runtime-зависимости, не переносим dev-инструменты.

## 3. Варианты упаковки (рекомендация)

1. **MSIX**
   - Плюсы: чистая установка/удаление, цифровая подпись, современный формат.
   - Минусы: ограничения по некоторым системным сценариям/legacy-интеграциям.

2. **WiX Toolset (MSI)** — рекомендуемый baseline для офиса
   - Плюсы: гибкий контроль prereq, enterprise-friendly, предсказуемая тихая установка.
   - Минусы: выше порог настройки.

3. **Inno Setup** — быстрый практичный старт
   - Плюсы: быстрый запуск, легко добавить проверку и установку зависимостей.
   - Минусы: меньше enterprise-стандартизации, чем MSI.

Рекомендация: если нужен быстрый запуск — Inno Setup; если долгосрочно и строго — WiX/MSI.

## 4. Минимальная логика инсталлятора

1. Проверить наличие .NET Runtime (или пропустить при self-contained publish).
2. Проверить наличие Ghostscript:
   - если нет — установить из пакета/тихого инсталлятора;
   - если есть, но версия ниже минимальной — предложить обновление.
3. Скопировать файлы клиента.
4. Записать базовую конфигурацию (`ApiBaseUrl`, путь логов и т.д.).
5. Создать ярлык запуска.
6. Первый старт клиента: проверка обновления по `update.xml`.

## 5. Что лучше сделать в коде клиента заранее

1. Экран/диалог "Проверка окружения" (runtime diagnostics):
   - видит API,
   - видит критичные внешние зависимости,
   - показывает понятную ошибку администратору.
2. Конфиг API URL через файл/переменную, а не захардкоженный IP.
3. Понятный лог старта (для диагностики установок на новых ПК).

## 6. Release-процесс (установка + обновления)

1. Сборка клиента (`dotnet publish`, выбранный режим).
2. Сборка инсталлятора (MSI/EXE) с prereq-проверками.
3. Тест установки на чистой машине в LAN.
4. Публикация ZIP/XML для автообновления (этап 4).
5. Раскатка инсталлятора только для новых ПК; существующие ПК обновляются через AutoUpdater.

## 7. Риски и контрмеры

1. Разные версии Ghostscript на ПК -> минимальная поддерживаемая версия + проверка при старте.
2. Неверный API URL после установки -> мастер конфигурации/проверка соединения на первом запуске.
3. Недостаток прав у пользователя -> документировать запуск от администратора для первичной установки.
4. Конфликт ручной установки и автообновления -> фиксировать правила (installer для первичной установки, updater для дальнейших релизов).

## 8. Definition of Done этапа 5

1. Есть воспроизводимый installer (MSI/EXE) с версионированием.
2. На чистом ПК приложение устанавливается и стартует без ручной докрутки файлов.
3. Проверка зависимостей (включая Ghostscript) работает.
4. После установки клиент успешно получает обновление через `update.xml`.
5. Есть короткая инструкция для админа/менеджера "как поставить на новый ПК".
