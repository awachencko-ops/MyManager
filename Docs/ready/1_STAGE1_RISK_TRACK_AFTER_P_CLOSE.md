# Этап 1: Risk Track (после закрытия P)

Дата: 2026-03-16  
Контекст: `P1...P5` этапа `single-first` закрыты.

## 1. Приоритетный порядок

1. `R2` Фильтр пользователей не связан с фактическими данными заказов.
2. `R3` Multi-order backend есть, UI пока single-only.
3. `R1` MainForm остается крупным aggregate.
4. `R4` Остаточные string-контракты.
5. `R5` Legacy path-слой order-level vs item-level.
6. `R8` LAN feature-gate не закреплен формально.

## 2. Минимальный план отработки

| Risk | Что делаем | Критерий закрытия |
|---|---|---|
| R2 | Привязать user-filter к фактическому полю/источнику пользователя в заказе и истории | Фильтр возвращает корректный набор строк на реальных данных без ложных совпадений |
| R3 | Поднять минимальный multi-order UI (group/item список + базовые item actions) | Multi-order сценарий выполняется без переключения на legacy |
| R1 | Продолжить разрезку MainForm на domain-компоненты/координаторы | MainForm уменьшается, бизнес-логика вынесена в отдельные классы |
| R4 | Дочистить string-контракты в оставшихся участках | Ключевые контракты централизованы в typed-константах/enum |
| R5 | Убрать неоднозначность путей, закрепить единый source-of-truth | Нет рассинхрона order/item путей в истории и UI |
| R8 | Ввести явный LAN feature-gate и поведение offline/online | Сетевой режим предсказуем, fallback формализован и документирован |

## 3. Режим фиксации

1. Каждое изменение по рискам фиксировать в `Docs/этапы/1_MAINFORM_MIGRATION_COMPLEX_RESEARCH_AND_PLAN.md`.
2. По каждому риску закрывать минимум один измеримый критерий.
3. После закрытия рисков переходить на этап 2 (`2_MULTI_ORDER_LOGIC_AND_POSTGRESQL_PLAN.md`).
