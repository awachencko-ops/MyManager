# План миграции таблицы: OLV production + Avalonia R&D (2026-04-02)

## Решение
1. Основной production-путь: `OLV` в текущем WinForms workspace.
2. Параллельный R&D-путь: `Avalonia` прототип для долгосрочного UI-этапа.

## Этап 0. Управление риском (до кодинга)
1. Ввести `feature-flag` (`UseOlvOrdersGrid`).
2. Зафиксировать baseline-метрики текущего `DataGridView`:
- `rebuild-grid elapsed`,
- фильтрация,
- субъективная плавность скролла.
3. Зафиксировать checklist паритета (клики, DnD, контекст, сортировка, selection, tiles sync).

Критерий выхода: baseline записан, rollback по флагу готов.

## Этап 1. Подготовка архитектуры (без смены UI)
1. Вынести `OrdersGridRowModel` и builder из `RebuildOrdersGrid`.
2. Вынести stage-операции в отдельный interaction-service.
3. Вынести фильтрацию/сортировку/counters из обхода UI-строк в модельный слой.

Критерий выхода: текущий DataGridView работает через новую модель, поведение не изменено.

## Этап 2. OLV внедрение под флагом
1. Добавить OLV-контрол в `pnlTable` как альтернативный контур.
2. Сделать `IOrdersGridAdapter` с двумя реализациями:
- `DataGridViewOrdersAdapter`,
- `OlvOrdersAdapter`.
3. Переключать адаптер через `UseOlvOrdersGrid`.

Критерий выхода: приложение стабильно стартует и в старом, и в новом режиме.

## Этап 3. Функциональный паритет OLV
1. Колонки и значения 1:1 к production таблице.
2. Group/item иерархия через `TreeListView`.
3. Клики/двойные клики/stage actions.
4. Контекстные команды.
5. Drag&Drop файлов.
6. Tooltip/hover/selection.
7. Сортировка + индикаторы.

Критерий выхода: parity matrix пройдена без критичных отклонений.

## Этап 4. Тесты и стабилизация
1. Обновить smoke/regression тесты с минимизацией прямой зависимости от приватного `dgvJobs`.
2. Добавить adapter-level тесты для list контуров.
3. Прогнать stress smoke на больших списках.

Критерий выхода: green тесты, нет P1/P2 регрессий.

## Этап 5. Rollout OLV
1. Включить OLV по умолчанию в dev/staging.
2. После стабилизации включить в production.
3. Старый DataGridView оставить как временный rollback до конца стабилизационного окна.

Критерий выхода: OLV по умолчанию, rollback не потребовался в agreed period.

## Этап 6. Cleanup
1. Удалить мертвые ветки DataGridView-only кода.
2. Упростить обработчики формы до adapter boundary.
3. Финализировать документацию по новому контуру.

Критерий выхода: один поддерживаемый production list-контур.

## Параллельный трек: Avalonia R&D
1. Сохранить отдельный проект `Prototypes/AvaloniaOrdersPrototype`.
2. Сравнивать с OLV по одинаковому сценарному набору.
3. Отдельно фиксировать:
- производительность,
- UX-оценку,
- стоимость переноса workflow,
- лицензионные ограничения.

## Decision Gate для возможного Avalonia production-cutover
Переоценивать решение только если одновременно выполнено:
1. OLV не закрывает KPI производительности/стабильности.
2. Avalonia прототип достиг функционального паритета.
3. Решен вопрос лицензирования/версий TreeDataGrid.
4. Подготовлен отдельный тестовый и rollout контур.

## Роллбек-план
1. До cleanup: мгновенный возврат на старый контур через `UseOlvOrdersGrid = false`.
2. При критичном инциденте в OLV: отключить флаг и вернуть DataGridView без hotfix-пересборки архитектуры.

## Что важно не пропустить
1. Нельзя делать миграцию только «заменой контрола в Designer».
2. Нельзя откладывать перенос логики из UI событий на конец проекта.
3. Нельзя смешивать production migration и экспериментальный Avalonia cutover в один релизный поток.
