# MASTER PLAN: Полный перенос MyManager на MainForm

Дата фиксации плана: 2026-03-10  
Период исполнения: 2026-03-09 — 2026-03-15 (следующая неделя)  
Главная цель: к концу недели перевести рабочий контур с legacy `Form1` на `MainForm` с подтвержденным паритетом ключевых сценариев.

## 1. Цель и Definition of Done

Полный перенос считается достигнутым, если одновременно выполнено:
1. Все сценарии основной таблицы работают из `MainForm` без обращения к legacy-обработчикам `Form1`.
2. Запуск/остановка обработки заказа, смена статусов и отображение логов выполняются из `MainForm`.
3. Контекстное меню, Drag&Drop и файловые операции по стадиям работают в `MainForm`.
4. Фильтры (`Состояние задания`, `Номер заказа`, `Пользователь`, `Дата поступления`, `Начало обработки`) стабильно работают без конфликтов popup-календарей/overlay.
5. Пройдена недельная регрессия на новой форме без блокирующих дефектов.

## 1.1 Критически важный принцип: group-first

1. Базовая модель всегда: `1 заказ = группа файлов (Items)`.
2. Схема `1 заказ = 1 файл` не отдельная архитектура, а частный случай group-first (ровно один `item`).
3. Для явного разделения вводится маркер:
   - `SingleFile` — один `item` в заказе.
   - `MultiFile` — два и более `item`.
4. Legacy-поля `SourcePath/PreparedPath/PrintPath` сохраняются только для обратной совместимости и синхронизируются с `item`-уровнем через нормализатор.

## 1.2 Конфликты и бесперебойное решение

| Потенциальный конфликт | Риск | Решение (принято) |
|---|---|---|
| В заказе есть пути на уровне order, но нет `Items` | Потеря файлов при переходе к групповой модели | Автонормализация: создается 1 synthetic `item` и пути переносятся в него |
| В single-заказе `order.PrintPath` и `item.PrintPath` расходятся | Непредсказуемая обработка/копирование | Правило приоритета: сначала существующий файл, затем непустой путь; после выбора значения синхронизируются |
| Разные части кода по-разному определяют «группу» | Случайные регрессии UI и обработки | Единый `OrderTopologyService.IsMultiFileOrder(...)` вместо локальных проверок `Items.Count > 1` |
| Маркер single/multi не совпадает с фактическим составом `Items` | Неверная маршрутизация сценариев | Нормализатор принудительно пересчитывает маркер при каждом сохранении/запуске |
| Старая история содержит null/битые item | Падения в grid/processor | Нормализатор очищает null-item, восстанавливает `ItemId`, `SequenceNo`, `UpdatedAt` |

## 6. Правила внедрения, отката и фиксации прогресса

1. Один день = один фокусный блок переноса (без смешивания крупных блоков).
2. После каждого блока обязательна ручная регрессия по чеклисту из раздела 5.
3. Любая блокирующая регрессия возвращает текущий блок в `In progress`.
4. До конца недели `Form1` остается в `Forms/Archive/` только как fallback, но тестирование ведется на `MainForm`.
5. Все статусы работ фиксируются только в этом документе (другие сопроводительные файлы не используются).
